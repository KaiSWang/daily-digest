## 問題一：GitHub Pages 顯示 404

### 症狀
Push 到 GitHub 後，瀏覽 `kaiswang.github.io/daily-digest/` 顯示 404 頁面。

### 根因
GitHub Pages 預設是**關閉的**。即使 repo 裡有 `.github/workflows/deploy.yml`，Pages 的 Source 設定必須手動到 Settings → Pages 選擇「GitHub Actions」才會啟用。沒有這一步，GitHub 不知道你要用 Actions 來部署靜態網站。

### 分析邏輯
1. 確認 repo 是 Public 且檔案結構正確（index.html 在根目錄）
2. 到 Actions 頁面發現 workflow 有跑但失敗
3. 查看失敗的 Action 日誌，錯誤訊息是：「Get Pages site failed. Please verify that the repository has Pages enabled and configured to build using GitHub Actions」
4. 這表明 Pages 功能本身沒有被啟用

### 處理方式
到 GitHub repo → Settings → Pages → Build and deployment → Source → 選擇 **GitHub Actions** → 儲存。

### 教訓
GitHub Pages 需要兩件事同時到位：(1) workflow 檔案存在 (2) Settings 裡啟用 Pages 並選對 Source。

## 問題二：GitHub Actions Deploy 失敗（Node.js 20 警告）

### 症狀
Actions 日誌顯示 2 個 error + 1 個 warning：
- Error: `HttpError: Not Found`
- Error: `Get Pages site failed`
- Warning: `Node.js 20 actions are deprecated`

### 根因
同問題一，Pages 未啟用。Node.js 20 警告是次要問題（GitHub 正在淘汰 Node 20 的 action runner），不影響功能，但未來可能需要升級 action 版本。

### 處理方式
啟用 Pages 後，回到 Actions 頁面 → 點進失敗的 run → Re-run all jobs。或是直接做一次 push 觸發新的 workflow run。

## 問題五：git push 被 reject（remote contains work you don't have）

### 症狀
```
! [rejected] main -> main (fetch first)
error: failed to push some refs
Updates were rejected because the remote contains work that you do not have locally.
```

### 根因
在你本地 commit 之前，GAS 已經透過 GitHub API 推了多筆 commit 到 remote（AI/GitHub 趨勢報告和 manifest 更新）。本地的 git history 落後於 remote，所以 push 被拒絕。

### 分析邏輯
1. 錯誤訊息明確說 remote 有本地沒有的 commit
2. 這是因為 GAS 的 `testGitHubPush` 或 `processAndForward` 已經在推檔案

### 處理方式
```bash
git pull --rebase origin main
```
這會把本地的 commit 「搬到」remote 最新 commit 的後面，保持線性歷史。

### 為什麼用 --rebase 而不是 merge
- `merge` 會產生一個額外的 merge commit，歷史比較亂
- `rebase` 讓歷史保持一條直線，更乾淨

## 問題六：git rebase 時出現 CONFLICT

### 症狀
```
CONFLICT (add/add): Merge conflict in data/ai-github/AI_GitHub趨勢_2026-03-26.md
CONFLICT (add/add): Merge conflict in data/ai-github/AI_GitHub趨勢_2026-03-27.md
CONFLICT (content): Merge conflict in data/manifest.json
```

### 根因
本地和 remote 都新增了相同路徑的檔案（AI/GitHub 趨勢報告），但內容不同：
- Remote 版本：GAS 從 Gmail 草稿擷取的完整報告內容
- 本地版本：我從 session transcripts 手動整理的摘要版本
- manifest.json：兩邊都更新了日期陣列，格式略有不同

### 分析邏輯
1. `add/add` conflict = 兩邊都新增了同名檔案
2. `content` conflict = 同一檔案兩邊都有修改
3. Remote 的 AI/GitHub 報告是 GAS 從草稿推上去的（較完整），應該保留 remote 版本

### 處理方式
```bash
# AI/GitHub 報告：保留 remote（GAS 推的完整版本）
git checkout --theirs "data/ai-github/AI_GitHub趨勢_2026-03-26.md"
git checkout --theirs "data/ai-github/AI_GitHub趨勢_2026-03-27.md"

# manifest.json：手動寫入正確內容（兩邊本質相同）
cat > data/manifest.json << 'EOF'
{ ... 正確的 JSON 內容 ... }
EOF

# 標記解決並繼續 rebase
git add -A
git rebase --continue
```

### 教訓
當有多個來源（本地 + GAS）同時推送到同一個 repo 時，容易產生衝突。解決方式：
1. 讓 GAS 成為唯一推送報告的管道（已完成）
2. 本地只負責推送網站設計變更（HTML/CSS）
3. 避免兩邊同時修改相同檔案

## 問題七：git index.lock 卡住

### 症狀
```
fatal: Unable to create '.git/index.lock': File exists.
Another git process seems to be running in this repository
```

### 根因
前一次 git 操作（可能是 Cowork 的 agent 和你的 terminal 同時操作同一個 repo）異常中斷，留下了 lock 檔案。Git 用 index.lock 防止多個 git 程序同時修改 index，如果程序異常退出，lock 不會自動清除。

### 處理方式
```bash
rm -f .git/index.lock
```
然後重新跑 git 指令即可。

### 注意
刪除前確認沒有其他 git 程序在跑。如果有多個 terminal 同時對同一個 repo 做 git 操作，要等前一個完成再跑下一個。
